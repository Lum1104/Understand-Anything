Understand-Anything Pipeline

Interactive walkthrough of /understand (seven phases), the auto-update hook (four phases), and every input and output along the way. Each step expands to show what it reads, what it writes, and which subagent or bundled script does the work.

Scope: skills/understand + hooks/auto-update-prompt.md Sources: skills/understand/SKILL.md, hooks/auto-update-prompt.md, agents/*.md Tip: click a node in the diagrams to jump; use the search box to filter steps.

/understand — main pipeline

Seven numbered phases plus a fractional 0.5. Phases 1, 3, 4, 5, and the optional --review path of 6 dispatch LLM subagents; phases 0, 0.5, 2 (merge), 6 (default), and 7 run bundled scripts deterministically. Click a phase node to jump.

Phase 0

Pre-flight

Resolve the project root, build the plugin if needed, capture project context, and choose between a full rebuild and an incremental update.

Parse $ARGUMENTS for a non-flag directory path, fall back to cwd. Verify the path exists. Worktree redirect: if the path is inside a git worktree (not the main checkout), redirect to the main repo root — worktrees are ephemeral and .understand-anything/ would be destroyed on session end (issue #133).

  • envUNDERSTAND_NO_WORKTREE_REDIRECT

Sets

Phase 1

SCAN — project inventory

Single dispatch of the project-scanner subagent, which runs bundled scripts to enumerate files, detect languages and frameworks, count lines, and build the import map.

1.1 Dispatch project-scanner agent LLM subagent

Orchestrator builds the dispatch prompt with $README_CONTENT, $MANIFEST_CONTENT, and $LANGUAGE_DIRECTIVE appended.

Calls

  • agentagents/project-scanner.md

Provides to agent

  • $PROJECT_ROOT
  • $README_CONTENT
  • $MANIFEST_CONTENT
  • $LANGUAGE_DIRECTIVE
  • output path: intermediate/scan-result.json
1.2 Subagent: discovery + categorization subagent + script

Subagent writes a discovery script (or invokes scan-project.mjs in current builds). Enumerates files via git ls-files -z when available, falling back to recursive walk. Applies default + .understandignore exclusions. Categorizes each file into code / config / docs / infra / data / script / markup.

Calls

  • node <SKILL_DIR>/scan-project.mjs $PROJECT_ROOT $OUT

Reads

  • cmdgit ls-files -z (or walker fallback)
  • fileevery source file (for line count)
  • file.understandignore, defaults

Reports

  • files list with {path, sizeLines, language, fileCategory}
  • detected languages, frameworks
  • complexity estimate
  • filteredByIgnore count
1.3 Subagent: build import map subagent + script

Per language, resolve relative imports for every code file. External package imports are dropped. The resulting map enables Phase 2 to construct cross-file batch context.

Calls

  • node <SKILL_DIR>/extract-import-map.mjs ...

Reads

  • fileevery fileCategory === "code" file's source
  • filetsconfig.json (for TS path aliases)
1.4 Write scan-result.json and return LLM subagent

Subagent writes a single artifact summarising the scan. Orchestrator reads it, stores importMap as $IMPORT_MAP for Phase 2 batch construction.

Writes

  • fileintermediate/scan-result.json

Orchestrator sets

  • $IMPORT_MAP, $FILE_LIST with fileCategory
>100-file gate: orchestrator pauses for user confirmation if the scan returned more than 100 files. Prompt-mediated.
Phase 2

ANALYZE — per-file node + edge extraction

The longest phase. Files are batched (semantic grouping via Louvain communities), file-analyzer subagents run in parallel waves, and a Python script merges + normalizes all batch outputs into one assembled graph.

2.1 Compute batches bundled script

Group files into batches of ~20–30 using Louvain community detection over the import graph, so related files end up in the same batch. Non-code files (Dockerfile + docker-compose, CI configs, etc.) are bundled together to enable cross-file edges.

Calls

  • node <SKILL_DIR>/compute-batches.mjs

Reads

  • fileintermediate/scan-result.json
  • var$IMPORT_MAP

Writes

  • batch plan (in memory or temp file)
  • per-batch batchImportData (neighbor symbols)
2.2 Dispatch file-analyzer subagents (up to 5 concurrent) LLM subagent ×N

For each batch, build a prompt including batchImportData + the file list with categories. Run up to 5 subagents at once; for N batches that's ⌈N/5⌉ sequential waves. Most of the wall-clock time for a large repo lives here.

Calls

  • agentagents/file-analyzer.md × N

Provides to each agent

  • batch index + total batches
  • batchImportData (pre-resolved imports)
  • files in batch with {path, sizeLines, language, fileCategory}
  • project name, languages, $LANGUAGE_DIRECTIVE
2.3 Each subagent: structural extraction subagent + script

The file-analyzer invokes extract-structure.mjs per file — tree-sitter parses the source and emits functions, classes, imports, exports. The agent then synthesizes GraphNode objects (with summaries, tags, complexity, languageNotes) and GraphEdge objects for that file.

Calls

  • node <SKILL_DIR>/extract-structure.mjs per file

Reads

  • each file's source code
  • tree-sitter WASM grammars (via web-tree-sitter)
2.4 Subagent: write batch output LLM subagent

Writes (per batch)

  • fileintermediate/batch-<N>.json with {nodes: [], edges: []}
2.5 Merge + normalize batches bundled script

After all batches return, a Python script reads every batch-*.json and produces one normalized graph. In one pass:

  • Combines all nodes/edges across batches
  • Normalizes node IDs (strips double prefixes, project-name prefixes, adds missing prefixes)
  • Normalizes complexity values (low→simple, medium→moderate, high→complex)
  • Rewrites edge refs to match corrected IDs
  • Deduplicates nodes by ID, edges by (source, target, type)
  • Drops dangling edges referencing missing nodes
  • Runs the tested_by linker (canonicalizes test-coverage edges, flips inverted ones, adds path-convention pairings, tags production nodes as tested)

Calls

  • python <SKILL_DIR>/merge-batch-graphs.py $PROJECT_ROOT

Reads

  • globintermediate/batch-*.json

Writes

  • fileintermediate/assembled-graph.json
  • stderrwarnings (appended to $PHASE_WARNINGS)
Incremental path: orchestrator first removes old nodes whose filePath matches a changed file and writes them as batch-existing.json. The merge script then combines them with fresh batches the same way.

ASSEMBLE REVIEW

A reviewer subagent reads the merged graph and cross-validates it against the import map, looking for missing cross-batch edges or obvious correctness issues.

3.1 Dispatch assemble-reviewer LLM subagent

Calls

Provides

  • fileintermediate/assembled-graph.json
  • globintermediate/batch-*.json
  • merge script stderr output

Writes

Orchestrator does

Phase 4

ARCHITECTURE — layer assignment

Identify architectural layers and assign every file-level node to exactly one layer. Each file-level node must end up in a layer.

4.1 Build language + framework + locale context orchestrator

For each detected language, read ./languages/<language-id>.md. For each framework, read ./frameworks/<framework-id>.md. If $OUTPUT_LANGUAGE != en, read ./locales/<lang>.md. These get appended to the agent prompt.

Reads

  • globskills/understand/languages/*.md
  • globskills/understand/frameworks/*.md
  • globskills/understand/locales/*.md (if non-en)

Sets

  • combined prompt template (in memory)
4.2 Dispatch architecture-analyzer LLM subagent

Single agent dispatch with the full file-level node list, all import edges, all other edges, and the directory tree. This is one of the two prompt-heavy phases (along with Phase 5).

Calls

  • agents/architecture-analyzer.md

Provides

  • all file-level nodes (id, type, name, filePath, summary, tags) — code + non-code
  • all import edges
  • all other edges (configures, documents, deploys, triggers, …)
  • directory tree
  • frameworks detected
  • language / framework / locale contexts (from 4.1)
  • previous layers (incremental only — for naming consistency)

Writes

  • fileintermediate/layers.json

Context-window hot spot

On large repos this agent's prompt can be heavy — the full file-level node set is materialized into a single prompt. Phase 4 (with Phase 5) is the most common context-overflow point in the pipeline.
4.3 Normalize layers.json orchestrator

Five-step normalization:

  1. Unwrap envelope: {layers: [...]} → [...]
  2. Rename legacy nodes field → nodeIds; extract .id from object entries
  3. Synthesize missing IDs as layer:<kebab-case-name>
  4. Convert raw file paths to typed IDs (file:<path>, config:<path>, etc.)
  5. Drop dangling nodeIds entries

Reads

  • fileintermediate/layers.json

Writes

  • fileintermediate/layers.json (normalized in place)
Phase 5

TOUR — guided learning path

Build a sequence of "tour steps" that walks a new reader through the codebase, starting from the entry point and aligning with the README narrative.

5.1 Dispatch tour-builder LLM subagent

Calls

  • agents/tour-builder.md

Provides

  • all file-level nodes (no function/class nodes)
  • layers (id, name, description; no nodeIds)
  • all edges (every type)
  • $README_CONTENT
  • $ENTRY_POINT
  • $LANGUAGE_DIRECTIVE

Writes

  • fileintermediate/tour.json
5.2 Normalize tour.json orchestrator

Five-step normalization mirroring 4.3:

  1. Unwrap {steps: [...]} → [...]
  2. Rename legacy nodesToInspect → nodeIds; whyItMatters → description
  3. Convert raw file paths to typed IDs
  4. Drop dangling refs
  5. Sort by order

Preserves optional languageLesson.

Phase 6

REVIEW — validation

Assemble the full KnowledgeGraph JSON and validate it. Two paths: deterministic inline check (default) or full LLM reviewer (--review flag).

6.1 Assemble final graph JSON orchestrator

Compose {version, project, nodes, edges, layers, tour} from the intermediate files.

Reads

  • fileintermediate/layers.json
  • fileintermediate/tour.json
  • var$COMMIT_HASH, project metadata

Writes

  • fileintermediate/assembled-graph.json (rewritten with full structure)
6.2a Default path: deterministic inline validator inline node script

Orchestrator writes a one-off Node.js validator to tmp/ua-inline-validate.cjs and executes it. Checks node IDs unique, edge refs resolve, every file-level node lives in exactly one layer, tour refs exist, etc.

Calls

  • node tmp/ua-inline-validate.cjs <graph> <output>

Writes

  • fileintermediate/review.json with {issues, warnings, stats}
6.2b --review path: LLM graph-reviewer LLM subagent

Dispatched only when --review is in $ARGUMENTS. Cross-validates graph against Phase 1 scan inventory — flags files in scan that have no corresponding node, or nodes referring to files not in scan.

Calls

  • agents/graph-reviewer.md

Writes

  • fileintermediate/review.json
6.3 Apply automated fixes orchestrator

If issues is non-empty: drop dangling edges, fill empty fields (tags → ["untagged"], summary → "No summary available"), remove nodes with invalid types. Re-run validator once. If issues persist, save anyway with warnings and skip dashboard auto-launch.

7.1 Write knowledge-graph.json orchestrator

Reads

Writes

  • file.understand-anything/knowledge-graph.json
7.2 Build fingerprint-input.json orchestrator

Heredoc-write a JSON file with {projectRoot, sourceFilePaths, gitCommitHash} for the fingerprint script to consume.

Reads

  • varsource file paths from Phase 1
  • var$COMMIT_HASH

Writes

  • fileintermediate/fingerprint-input.json
7.3 Generate fingerprint baseline bundled script

Tree-sitter extracts functions, classes, imports, exports per source file. Writes the structural baseline used by every subsequent auto-update.

Calls

  • node <SKILL_DIR>/build-fingerprints.mjs <input.json>

Reads

  • fileintermediate/fingerprint-input.json
  • fileevery source file (for hash + structural extraction)
  • module@understand-anything/core (TreeSitterPlugin)

Writes

  • file.understand-anything/fingerprints.json
  • stdoutFingerprints baseline: <N> files

Critical — must succeed before step 7.4

SKILL.md instructs: if the script exits non-zero or stdout doesn't contain Fingerprints baseline:, abort Phase 7 and do not proceed to step 7.4. Otherwise meta.json advances without a valid baseline and every subsequent auto-update sees "no fingerprint" for every file, classifying them STRUCTURAL → permanent FULL_UPDATE cascade (issue #152 family).
7.4 Write meta.json orchestrator

Only after step 7.3 succeeded. Records the current commit hash, timestamp, version, and analyzed file count.

Reads

  • var$COMMIT_HASH
  • varanalyzedFiles count

Writes

  • file.understand-anything/meta.json
7.5 Clean up intermediates orchestrator

Removes

  • dir.understand-anything/intermediate/
  • dir.understand-anything/tmp/
Interrupted runs leak. If Phase 7 errored before this step, intermediate/ survives. The next /understand --full will scan back into it because .understand-anything/ isn't in DEFAULT_IGNORE_PATTERNS — see failure modes.
7.6 Report summary orchestrator

Print: project name, files analyzed by fileCategory, nodes by type, edges by type, layers, tour step count, warnings, output paths.

7.7 Auto-launch dashboard (conditional) orchestrator

Invokes the /understand-dashboard skill, but only if final validation passed after normalization. If validation failed, dashboard launch is skipped and the user is told.

Auto-update hook

Triggered by the PostToolUse hook on git commit/merge/cherry-pick/rebase, and by SessionStart when meta.json's commit hash differs from HEAD. Requires autoUpdate: true in config.json. Goal: zero LLM tokens when changes are cosmetic.

AU Phase 1

Structural fingerprint check — zero LLM tokens

A Node script compares each changed file against its stored fingerprint and classifies the overall change into one of four buckets.

AU.1.1 Per-file classification inline node script
ComparisonClass
SHA-256 of file content matches stored hashNONE — skip entirely
Hash differs, regex-extracted functions / classes / imports / exports identicalCOSMETIC
Hash differs, structural elements differSTRUCTURAL
File new (not in fingerprints.json)STRUCTURAL
File deleted (in fingerprints, missing on disk)STRUCTURAL

Reads

  • file.understand-anything/fingerprints.json
  • each changed file's source

Writes

  • fileintermediate/change-analysis.json
AU.1.2 Aggregate decision orchestrator
ConditionAction
All NONE / COSMETICSKIP — bump meta.json, STOP
Some STRUCTURAL, ≤10 files, same dirsPARTIAL_UPDATE → Phase 2
New/deleted dirs or >10 structural filesARCHITECTURE_UPDATE → Phase 2 + flag
>30 structural files or >50% of graphFULL_UPDATE — recommend manual /understand --full, STOP

Why this gate: the hook silently aborting on FULL_UPDATE is intentional — escalating from auto-update would burn lots of tokens without user consent.

AU Phase 2

Targeted re-analysis

Only this phase costs LLM tokens. Dispatches file-analyzer subagents on just the structurally-changed files, then surgically merges results into the existing graph.

AU.2.1 Read existing graph + batch the changed files orchestrator

≤10 files: one batch. Otherwise batches of 5–10.

Reads

  • file.understand-anything/knowledge-graph.json
  • $CHANGED_FILES (structural only)
AU.2.2 Dispatch file-analyzer on changed files only LLM subagent

Reuses the same agent definition as the full pipeline's Phase 2, with an explicit instruction not to invent nodes for files outside the batch.

Calls

  • agents/file-analyzer.md

Writes

  • fileintermediate/batch-<N>.json per batch
AU.2.3 Surgical merge into existing graph orchestrator
  • Remove old nodes whose filePath matches any reanalyzed file or any deleted file
  • Remove edges referencing removed nodes
  • Add new nodes/edges from the fresh batches
  • Deduplicate nodes by ID, edges by (source, target, type)
  • Drop any remaining dangling edges
AU Phase 3

Conditional architecture + save

Update layers as needed, save the graph, update meta, and most importantly LOAD-PATCH-SAVE the fingerprints (don't overwrite).

AU.3a Architecture update (ARCHITECTURE_UPDATE only) LLM subagent

Re-dispatch architecture-analyzer with previous layer definitions injected for naming consistency.

AU.3b Lite layer update (PARTIAL_UPDATE only) orchestrator
  • For new files: assign to the most likely existing layer by directory path match
  • For deleted files: remove their IDs from layer nodeIds
  • Remove any layer that ends up empty
AU.3c Lite validation orchestrator
  • Drop dangling edge refs
  • Drop dangling layer nodeIds
  • Ensure every file node lives in exactly one layer (catch-all if missing)
AU.3d Save graph + meta + LOAD-PATCH-SAVE fingerprints inline node script

The most subtle step. Inline script must:

  1. LOAD ALL existing entries from fingerprints.json. Never skip.
  2. PATCH only re-analyzed paths (compute new hash + structural elements).
  3. REMOVE entries for deleted files.
  4. SAVE ALL entries back as one dict.
  5. GUARD: if fingerprints.json existed and was non-empty but loaded as {}, abort — silent load failure would otherwise clobber the whole baseline (the issue #152 reproduction).

Reads

  • filefingerprints.json (entire dict)
  • each reanalyzed file's source

Writes

  • fileknowledge-graph.json
  • filemeta.json
  • filefingerprints.json (full dict, not just patched subset)

LOAD-PATCH-SAVE invariant

The original #152 reproduction wrote only the freshly-computed batch entries to fingerprints.json, discarding everything else. The next auto-update saw all those as "new", classified them STRUCTURAL, and escalated to FULL_UPDATE forever. The 2.7.3 fix made the LOAD-PATCH-SAVE script explicit and added the existedAndNonEmpty && before === 0 guard.

Failure modes

The most common ways the pipeline corrupts itself, and where to look first when something is off.

Silent fingerprints failure (issue #152 family)

Symptom: meta.json + knowledge-graph.json present, fingerprints.json absent. Auto-update permanently broken — every commit triggers FULL_UPDATE because every file has no stored fingerprint to compare against.

The 2.7.3 fix (commits dd8b724 + e7af9ae) addressed the original PARTIAL_UPDATE overwrite case and added a prompt-level "abort Phase 7 if fingerprints fail" instruction. A residual gap remains when the orchestrator (the LLM) proceeds to write meta.json despite the script failing — typically because node isn't reachable in the non-interactive Bash subshell the skill runs in (a common environment issue with version managers like nvm/fnm/mise hooked into ~/.zshrc instead of ~/.zshenv).

If you see this symptom, manually invoke the bundled script against an input built from your repo's source paths — the script's actual error will tell you the underlying cause:

node skills/understand/build-fingerprints.mjs /tmp/fp-input.json

Intermediate-directory pollution

If a run errors out before Phase 7.5, leftover intermediate/batch-*.json and related files survive. The next /understand --full walks into them because .understand-anything/ is not in DEFAULT_IGNORE_PATTERNS (packages/core/src/ignore-filter.ts:9) — so file counts grow monotonically across runs.

Workaround: add .understand-anything/ to your .understandignore, or manually delete intermediate/ between runs if a previous one failed.

Symlink resolution failure (issue #162)

Fixed in 2.7.3. extract-structure.mjs's isCli guard compared import.meta.url (resolved through symlinks) against process.argv[1] (preserved as symlink). When SKILL_DIR was a symlink (the default Claude Code install layout), main() never ran. Now uses realpathSync on both sides.

Subagent dispatch overhead dominates wall-clock time

Each subagent cold-start is 30–60s regardless of payload. For 700-file repos, Phase 2 runs ~28 batches in waves of 5 — Phase 2 alone is typically 60–80% of total time. The bottleneck is LLM throughput; raising the concurrency cap (currently 5 in SKILL.md) gives roughly proportional speedup.
