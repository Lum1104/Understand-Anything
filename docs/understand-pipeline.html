1. Overview

Understand-Anything analyzes a codebase to produce a knowledge graph — a JSON file at .understand-anything/knowledge-graph.json that the React dashboard renders as an interactive view of the project's architecture.

The pipeline has two execution paths:

  • Full analysis — the user invokes /understand (with or without --full). Runs all seven phases. Takes minutes to over an hour depending on repo size.
  • Auto-update — triggered by a git commit hook when the user has opted in via --auto-update. Uses structural fingerprints to detect what changed and re-analyzes only the affected files. Spends zero LLM tokens when changes are cosmetic.

2. The main pipeline: /understand

Seven numbered phases plus a fractional 0.5. Each phase reports its progress to the user with a [Phase N/7] status line (per SKILL.md:25–38). Phases 1, 3, 4, 5, 6 (LLM path) dispatch subagents defined in agents/; Phases 2 and 6 (deterministic path) and 7 run bundled scripts.

Phase 0

Pre-flight

Resolve project root, build the plugin if needed, decide between full and incremental, gather context for later subagents.

reads existing knowledge-graph.json + meta.json if present, config.json, README, package manifest

produces in-memory: PROJECT_ROOT, PLUGIN_ROOT, $IMPORT_MAP, $DIR_TREE, $ENTRY_POINT, decision (FULL / INCREMENTAL / REVIEW / SKIP)

Sub-steps

  1. Resolve PROJECT_ROOT — parse $ARGUMENTS for a directory path or default to cwd. Redirect git worktrees back to the main checkout (worktrees are ephemeral and would destroy .understand-anything/ on session end — issue #133).
  2. Ensure the plugin is built — locate PLUGIN_ROOT (multiple candidate paths supported: Claude, Copilot, Codex, opencode, pi, plain clone), then build packages/core if its dist/ is missing. If pnpm is missing the skill instructs the user to install Node.js ≥ 22 and pnpm ≥ 10.
  3. Get the current git commit hash
  4. Create .understand-anything/intermediate and .understand-anything/tmp directories
  5. Apply --auto-update / --no-auto-update flags — writes to .understand-anything/config.json
  6. Apply --language flag — sets $OUTPUT_LANGUAGE and stores in config.json for cross-run consistency
  7. Merge any subdomain graphs — runs merge-subdomain-graphs.py if files like frontend-knowledge-graph.json exist alongside the main graph
  8. Decision logic — full rebuild, incremental update, review-only, or skip:
    ConditionAction
    --full in argumentsFull analysis (all phases)
    No existing graph or metaFull analysis (all phases)
    --review + unchanged commit hashSkip to Phase 6 review-only
    Existing graph + unchanged commit hashAsk the user
    Existing graph + changed filesIncremental update (Phase 2 only on changed files)
  9. Collect project context — README (first 3000 chars), package manifest, directory tree (top 2 levels), entry-point detection
Phase 0.5

Ignore configuration

Generate or confirm .understandignore patterns. Uses .gitignore as a starting point.

reads .understand-anything/.understandignore, .gitignore

produces .understand-anything/.understandignore (if absent)

Confirmation gate. The skill pauses here waiting for user confirmation. This is a prompt-mediated pause, not enforced — explicit "proceed without confirmation" in the invocation skips it.
Phase 1

SCAN — project inventory

Enumerate every file, detect languages and frameworks, build an internal-import map. Single subagent dispatch.

dispatches agents/project-scanner.md

script skills/understand/scan-project.mjs (bundled; runs as Node subprocess)

script skills/understand/extract-import-map.mjs (bundled; per-language import resolution)

produces .understand-anything/intermediate/scan-result.json — files list with fileCategory, languages, frameworks, complexity estimate, importMap

Gate at >100 files. Skill asks for user confirmation. Prompt-mediated; override in invocation if running unattended.
Phase 2

ANALYZE — per-file node and edge extraction

Batch files into groups of ~20–30, dispatch subagents in parallel (up to 5 concurrent) to extract GraphNode and GraphEdge objects per file.

dispatches agents/file-analyzer.md (N batches × up to 5 concurrent)

script skills/understand/compute-batches.mjs (semantic batching via Louvain communities since 2.7.x)

script skills/understand/extract-structure.mjs (tree-sitter pass per file)

script skills/understand/merge-batch-graphs.py (merges all batch-*.json + normalizes IDs)

produces intermediate/batch-N.json (one per batch), intermediate/assembled-graph.json (merged + normalized)

What the merge script does

  • Combines nodes/edges across batches
  • Normalizes node IDs (strips double prefixes, project-name prefixes, adds missing prefixes)
  • Normalizes complexity values (low→simple, medium→moderate, high→complex)
  • Deduplicates nodes by ID and edges by (source, target, type)
  • Drops dangling edges referencing missing nodes
  • Runs the tested_by linker — flips inverted test→production edges and adds path-convention pairings (X.ts ↔ X.test.ts, Java/Maven src/test/... ↔ src/main/..., etc.)

Incremental path

On incremental update, only changed files are batched. Old nodes whose filePath matches a changed file are removed from the prior graph, then merged with the fresh batches.

Phase 3

ASSEMBLE REVIEW

A reviewer subagent checks the assembled graph against the import map for missing cross-batch edges and obvious correctness issues.

dispatches agents/assemble-reviewer.md

reads intermediate/assembled-graph.json, all batch-*.json, $IMPORT_MAP

produces intermediate/assemble-review.json (notes appended to $PHASE_WARNINGS)

Phase 4

ARCHITECTURE — layer assignment

Identify architectural layers and assign every file-level node to exactly one layer.

dispatches agents/architecture-analyzer.md — receives the full file-level node list, all import edges, and all other edges in the dispatch prompt

produces intermediate/layers.json

Context-window hot spot

On large repos this agent's prompt can get heavy — the full node list is materialized into one prompt. Phase 4 (with Phase 5) is the most common context-overflow point in the pipeline.

Normalization after the subagent

The orchestrator unwraps any { "layers": [...] } envelope, renames legacy fields (nodes → nodeIds), synthesizes missing IDs (layer:<kebab-case-name>), converts raw file paths to typed IDs (file:<path>, config:<path>, etc.), and drops dangling references.

Phase 5

TOUR — guided learning path

Build a sequence of "tour steps" that lead a new reader through the codebase, anchored to the entry point and aligned with the README narrative.

dispatches agents/tour-builder.md — receives all file-level nodes, layers (without nodeIds), and all edges

produces intermediate/tour.json (normalized to {order, title, description, nodeIds, languageLesson?})

Phase 6

REVIEW — validation

Assemble the final KnowledgeGraph object and validate it. Two paths.

script tmp/ua-inline-validate.cjs (default: deterministic) — checks node IDs, edge refs, layer membership, tour refs; produces intermediate/review.json

dispatches agents/graph-reviewer.md (only with --review flag)

Automated fixes for non-empty issues array — drop dangling edges, fill empty fields, remove invalid nodes; re-validate once.

Phase 7

SAVE — persist + baseline

Write the final graph, generate the fingerprint baseline, persist metadata, clean up intermediates. Step ordering is critical here.

Steps

  1. Write knowledge-graph.json — the user-facing artifact.
  2. Generate the fingerprint baseline — runs node skills/understand/build-fingerprints.mjs. Tree-sitter extracts functions/classes/imports/exports per source file and writes .understand-anything/fingerprints.json. If this fails, Phase 7 must abort — meta.json must not advance.
  3. Write meta.json — records the current commit hash, last-analyzed timestamp, version, file count.
  4. Clean up — rm -rf intermediate/ tmp/.
  5. Report summary — phase warnings, node/edge counts by type, output paths.
  6. Auto-launch dashboard (only if final validation passed) — invokes /understand-dashboard.

Critical ordering — fingerprints before meta.json

If meta.json records a new commit hash but fingerprints.json never got written, every subsequent auto-update sees "no stored fingerprint" for every file, classifies them all as STRUCTURAL, and escalates to FULL_UPDATE on every commit. This is the issue #152 family of bugs. The 2.7.3 fix added the prompt-level "abort" instruction; the 2.7.5+ preflight guards against the upstream cause (missing node on PATH).

3. The auto-update pipeline

Triggered by the PostToolUse hook on git commit/merge/cherry-pick/rebase, plus the SessionStart hook when meta.json's commit hash differs from HEAD. Both require autoUpdate: true in .understand-anything/config.json.

Goal: spend zero LLM tokens when changes are cosmetic; only invoke LLM agents when structural changes are detected. Lives in hooks/auto-update-prompt.md.

Phase 0

Pre-flight + filter

  1. Verify knowledge-graph.json and meta.json exist; if not, instruct user to run /understand first.
  2. Get current commit hash; if it matches stored, report up-to-date and stop.
  3. git diff <lastCommitHash>..HEAD --name-only — get changed files.
  4. Filter to source-file extensions only.
  5. Apply .understandignore exclusions via createIgnoreFilter (added in #153 fix).
  6. If nothing remains, just update meta.json commit hash and stop.
Phase 1

Structural fingerprint check — zero LLM tokens

A Node.js script compares each changed file against its stored fingerprint and classifies it.

Per-file classification

ComparisonClass
Content hash matches storedNONE (skip)
Hash differs, but functions/classes/imports/exports identicalCOSMETIC
Hash differs, structural elements differSTRUCTURAL
File new (not in fingerprints.json)STRUCTURAL
File deleted (in fingerprints, missing on disk)STRUCTURAL

Overall action

ConditionAction
All NONE/COSMETICSKIP — bump meta.json, stop
Some STRUCTURAL, ≤10 files, same directoriesPARTIAL_UPDATE
New/deleted directories or >10 structural filesARCHITECTURE_UPDATE
>30 structural files or >50% of graphFULL_UPDATE — recommend manual /understand --full
Phase 2

Targeted re-analysis

Only this phase costs LLM tokens. Dispatches file-analyzer subagents on just the structurally-changed files.

dispatches agents/file-analyzer.md on the structural change set only

Merges results into the existing graph: removes old nodes by filePath, removes edges referencing removed nodes, adds fresh nodes + edges, dedupes.

Phase 3

Conditional architecture/tour + save

  • 3a. Architecture update — only if ARCHITECTURE_UPDATE. Re-runs architecture-analyzer with the prior layer definitions injected for naming consistency.
  • 3b. Lite layer update — for PARTIAL_UPDATE. Directory-match new files into existing layers, drop deleted-file IDs.
  • 3c. Lite validation — drop dangling edges, ensure every file node is in exactly one layer.
  • 3d. Save — write graph, update meta.json, LOAD-PATCH-SAVE fingerprints.json (load all entries, patch only re-analyzed paths, save full dict back — the issue #152 guard).
  • Clean up intermediate/.

LOAD-PATCH-SAVE invariant

The script that updates fingerprints.json in Phase 3d must load every existing entry, patch only the re-analyzed paths, and save the full dict. Overwriting with just the patched subset corrupts the baseline so every subsequent commit escalates to FULL_UPDATE — the original #152 reproduction. The script has an explicit existedAndNonEmpty && before === 0 guard that refuses to overwrite if the load step silently returned empty.

5. Failure modes

Silent fingerprints failure (issue #152 family)

Symptom: meta.json + knowledge-graph.json present, fingerprints.json absent. Auto-update permanently broken — every commit triggers FULL_UPDATE because every file has "no stored fingerprint" to compare against.

The 2.7.3 fix (commits dd8b724 + e7af9ae) addressed the original PARTIAL_UPDATE overwrite case and added a prompt-level "abort Phase 7 if fingerprints fail" instruction. A residual gap remains when the orchestrator (the LLM) proceeds to write meta.json despite the script failing — typically because node isn't reachable in the non-interactive Bash subshell the skill runs in (a common environment issue with version managers like nvm/fnm/mise hooked into ~/.zshrc instead of ~/.zshenv).

If you see this symptom, manually invoke node skills/understand/build-fingerprints.mjs against an input file built from your repo's source paths — the script's actual error will tell you the underlying cause.

Intermediate-directory pollution

If a run errors out before Phase 7 step 4 cleans up, leftover intermediate/batch-*.json and related files survive. The next /understand --full walks back into them because .understand-anything/ is not in DEFAULT_IGNORE_PATTERNS (see packages/core/src/ignore-filter.ts:9) — so file counts grow monotonically across runs.

Workaround: add .understand-anything/ to your .understandignore, or manually delete intermediate/ between runs if a previous one failed.

Symlink resolution failure (issue #162)

Fixed in 2.7.3. extract-structure.mjs's isCli guard compared import.meta.url (resolved through symlinks) against process.argv[1] (preserved as symlink). When SKILL_DIR was symlinked (the default Claude Code install layout), main() never ran. Now uses realpathSync on both sides.

Subagent dispatch overhead

Each subagent cold-start is 30–60s regardless of payload. For 700-file repos, Phase 2 runs ~28 batches sequentially in waves of 5 — Phase 2 alone is typically 60–80% of total wall-clock time. The bottleneck is LLM throughput; raising the concurrency cap (currently 5 in SKILL.md:297) gives roughly proportional speedup.

6. Tunables and options

FlagEffect
--fullForce a full rebuild, ignoring any existing graph and skipping the Phase 0 decision question
--auto-updateEnable commit-triggered auto-update (writes autoUpdate: true to config.json); analysis still proceeds normally
--no-auto-updateDisable auto-update
--reviewRun the full LLM graph-reviewer in Phase 6 instead of the inline deterministic validator
--language <lang>Generate all textual content (summaries, descriptions, tags, titles) in the specified language. ISO 639-1 codes or friendly names accepted. Stored in config.json for cross-run consistency.
positional pathAnalyze the given directory instead of cwd (e.g. /understand ../other-project)

Other relevant config

  • .understand-anything/config.json — autoUpdate (boolean), outputLanguage (string)
  • UNDERSTAND_NO_WORKTREE_REDIRECT=1 — opt out of the worktree-to-main-checkout redirect in Phase 0
  • CLAUDE_PLUGIN_ROOT — explicit plugin root path (set by Claude Code automatically)
